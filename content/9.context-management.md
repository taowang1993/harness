---
title: "Context Management"
description: "How long transcripts are compacted, persisted, pruned, and recalled across sessions."
---

Coding agents accumulate long transcripts: user prompts, assistant replies, tool calls, tool results, summaries, retries, branch changes, and more.

## Codex: `ContextManager`

Codex stores typed history and uses ghost snapshots when compacting.

## Claude Code: Multi-Strategy Compaction

Claude Code has several compaction modes and aggressive recovery logic.

## OpenCode: Session History + Hidden Summary Agents + Pruning

OpenCode persists sessions and uses multiple mechanisms to control context growth.

### Characteristics

- sessions can have **parent/child** relationships
- hidden system agents generate **titles**, **summaries**, and **compactions**
- when context gets tight, OpenCode can create a compaction task and continue from the summary
- after the loop, older tool outputs can be **pruned** to reduce future token pressure
- snapshots and diffs support undo/revert behavior separately from compaction

### Distinctive trait

OpenCode treats context management as both a runtime concern and a product workflow concern: compaction, summaries, child sessions, and revertable file changes all sit in the same session system.

## Pi: JSONL Session Tree + Branch Summaries

Pi persists context as a JSONL tree and makes branch summaries a first-class part of the session model.

## Hermes: SQLite Sessions + Dual Compression Layers

Hermes treats context as a combination of persistent session storage and pluggable compaction.

### Characteristics

- sessions are stored in **SQLite** with FTS5 full-text search
- session lineage is tracked with `parent_session_id`
- the default `ContextCompressor` does lossy summarization of middle turns
- a second **gateway hygiene** layer can auto-compress large sessions before the next agent run
- context management is pluggable through a **ContextEngine** interface
- `session_search` turns past sessions into part of the live recall system

### Distinctive trait

Hermes separates **conversation storage**, **compression policy**, and **cross-session recall** more explicitly than the other systems. It is less branch-centric than Pi and less undo-centric than OpenCode, but more operationally persistent than both.

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Main compaction idea** | ghost snapshot | compaction cascade | hidden compaction agent + pruning | persisted compaction entries | built-in compressor + gateway hygiene + pluggable engines |
| **Session branching** | resume/fork | resume/fork/rewind | child sessions and forks | tree navigation in one file | session lineage + ACP/session forks |
| **Separate summary agents** | No | Internal services | Yes | Extension/custom summary hooks | Auxiliary summary calls, not a dedicated visible agent |
| **Undo/revert tied to session model** | Some | Some | Strongly | More limited / file-system oriented | Limited |
| **Branch summaries** | Not central | Not central | Session/summary oriented | First-class | Not central |

## Takeaway

OpenCode sits between Claude Code and Pi again:
- more productized than Pi
- more open and inspectable than Claude Code
- more session-workflow-aware than Codex

Hermes sits somewhere else:
- more operationally persistent than Pi
- less UI-workflow-driven than OpenCode
- more focused on long-lived session storage, recall, and pluggable compaction engines.
