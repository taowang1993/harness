Searching for files and source text is one of the most common coding-agent operations.

## The Search Styles

| Approach | Agent | Shape |
|----------|-------|-------|
| Shell-delegated | Codex | Search is mostly a shell skill |
| Built-in wrappers | Claude Code | Search is a product feature |
| Rich built-ins | OpenCode | Search is a product feature with extra code-intel surfaces |
| Compact built-ins | Pi | Search is a small replaceable toolkit |
| Structured file tools | Hermes | Search is a dedicated file-tool layer, not something the model is supposed to improvise with shell |

## Codex: Shell-Delegated Search

Codex tells the model to prefer `rg`, but search still happens through shell commands.

## Claude Code: `GlobTool` + `GrepTool`

Claude Code wraps ripgrep in dedicated tools with structured parameters and bundled binaries.

## OpenCode: `grep`, `glob`, `list`, `codesearch`, and LSP

OpenCode gives the model a stronger built-in search surface than either Codex or Pi.

### Built-ins

- `grep` — regex content search via ripgrep
- `glob` — file pattern matching
- `list` — directory listing/tree-style browsing
- `codesearch` — hosted code search in some setups
- optional `lsp` — semantic navigation and symbols

### Characteristics

- search is permission-controlled
- `.gitignore` is respected by default, with `.ignore` escape hatches
- output is truncated and shaped for agent use
- OpenCode can blend text search, semantic LSP queries, and remote search tools in one product

## Pi: `grep`, `find`, `ls`

Pi provides a compact built-in search trio:

- `grep`
- `find`
- `ls`

They are small, typed, and replaceable through custom operations.

## Hermes: `search_files` + `read_file`

Hermes strongly nudges the model away from shell search patterns and toward dedicated file tools.

### Built-ins

- `search_files` — ripgrep-backed file-name and content search
- `read_file` — paginated, line-numbered file reading
- `terminal` is still available, but the tool guidance explicitly says **do not use cat/head/tail/grep/find when file tools exist**

### Characteristics

- search and read live together inside the `file` toolset
- results are structured for agent consumption rather than raw shell dumps
- the same tool surface works across local, container, SSH, and cloud terminal backends
- Hermes also separates **session search** (`session_search`) from source-file search, keeping code lookup distinct from conversational recall

## Comparison Table

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Search surface** | Shell | `GlobTool` + `GrepTool` | `grep` + `glob` + `list` + optional `lsp`/`codesearch` | `grep` + `find` + `ls` | `search_files` + `read_file` |
| **Depends on host tools?** | Yes | Less so | Uses internal/bundled helpers and product integrations | Can ensure helper tools / override backends | Often ripgrep-backed, but exposed through stable tools |
| **Semantic code intel** | Minimal | LSP tool | Built-in LSP integration | Extension/SDK territory | Minimal in core search layer |
| **Search as product feature?** | No | Yes | Yes | Lightly | Yes |

## Takeaway

Codex treats search as shell composition.
Claude Code and OpenCode treat search as a first-class product capability.
Pi treats search as a small built-in layer that you can keep or replace.
Hermes also treats search as a first-class tool surface, but frames it as part of a broader file-tool discipline rather than a code-intel subsystem.
