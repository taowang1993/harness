---
title: "Tool System"
description: "How coding agents define, expose, extend, and safeguard their tool surfaces."
---

Tools are how coding agents interact with the world: reading files, running commands, editing code, asking questions, searching docs, and sometimes spawning sub-agents.

## Codex: Few Powerful Tools

Codex keeps the core surface small and relies heavily on `shell` plus `apply_patch`.

## Claude Code: Many Specialized Tools

Claude Code exposes many purpose-built tools, each with its own schema, permission logic, and product UX.

## OpenCode: Rich Built-Ins + Plugins + MCP

OpenCode combines a substantial built-in tool set with multiple extension paths.

### Built-in tools

Core built-ins include:

- `bash`
- `read`
- `glob`
- `grep`
- `list`
- `edit`
- `write`
- `apply_patch`
- `multiedit`
- `task`
- `todowrite`
- `webfetch`
- `websearch`
- `codesearch`
- `skill`
- `question`
- optional `lsp`

OpenCode also conditionally prefers `apply_patch` vs `edit`/`write` depending on model/provider characteristics.

### Extension paths

OpenCode can be extended through:

- **plugins** with lifecycle hooks like `tool.execute.before` and `tool.execute.after`
- **custom tools** in `.opencode/tools/`
- **MCP servers**
- **agents** that can invoke subtasks through the `task` tool

### Distinctive feature

OpenCode is the most "product-like" open-source tool system in this comparison: many built-ins, built-in permissions, plugins, MCP, and custom tools all live in one coherent runtime.

## Pi: Small Core Toolkit, First-Class Public Extension Surface

Pi ships a smaller built-in set and makes custom tool registration a primary API.

### Built-ins

Default tools:
- `read`
- `write`
- `edit`
- `bash`

Optional built-ins:
- `grep`
- `find`
- `ls`

### Extension story

Pi tools can be registered from extensions or the SDK, override built-ins, and provide custom rendering.

## Hermes: Broad Registry + Toolsets + Backends

Hermes has the broadest built-in tool registry in this group, but it organizes that breadth through **toolsets**.

### Built-ins

Representative built-ins include:

- terminal and process tools: `terminal`, `process`
- file tools: `read_file`, `write_file`, `patch`, `search_files`
- web and browser tools: `web_search`, `web_extract`, `browser_*`
- orchestration tools: `todo`, `clarify`, `execute_code`, `delegate_task`
- persistence tools: `memory`, `session_search`
- automation/integration tools: `cronjob`, `send_message`, MCP tool wrappers, Home Assistant tools, RL tools

### Extension paths

Hermes can be extended through:

- **self-registering tool modules** in the repo
- **toolsets** that expose different bundles per platform
- **MCP tools** discovered dynamically from configured servers
- **plugins** that register new tools
- **memory providers** and **context engines** that inject additional behaviors and sometimes their own tool surfaces

### Distinctive feature

Hermes combines a large registry with operational concerns: platform-specific toolsets, approval flows, multiple terminal backends, and dynamic plugin/MCP discovery.

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Default tool count** | Small | Large | Large | Small-to-medium | Large |
| **Primary philosophy** | Shell-first | Specialized-tool-first | Product-rich + plugin/MCP extensible | Minimal built-ins + extension-first | Broad built-ins organized by toolsets and runtimes |
| **Custom tools** | Some dynamic tools | Some integrations | Plugins, custom tools, MCP | Extensions and SDK | Plugins, MCP, provider plugins |
| **Can override built-ins?** | Limited | Not central | Yes | Yes | Sometimes, via later registry/plugin registration |
| **Custom rendering/hooks** | Limited | Product-owned | Plugin hooks | Public render and event hooks | Callbacks, plugins, and platform-specific tool availability |

## Tradeoffs

- **Codex** keeps the system small by trusting the shell.
- **Claude Code** builds a large specialized tool ecosystem inside the product.
- **OpenCode** does something similar, but as an open-source, provider-agnostic platform with plugins and MCP.
- **Pi** keeps the core smaller and expects users to build the workflow they want.
- **Hermes** ships a much broader default surface, but relies on toolsets and runtime configuration so that breadth stays manageable.
