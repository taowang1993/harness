The agent loop is the heartbeat of a coding agent. It receives input, calls the model, executes tools, feeds results back, and repeats until the job is done.

## Codex: Turn-Based State Machine

Codex keeps explicit turn boundaries.

```text
User Input
    ↓
TurnContext creation
    ↓
Build prompt
    ↓
Stream response
    ↓
Extract tool calls
    ↓
Execute tools
    ↓
Append results
    ↓
Next turn if needed
```

## Claude Code: Generator-Based Streaming Loop

Claude Code uses an async generator and streams results continuously.

```text
User Input
    ↓
Assemble prompt
    ↓
Stream Anthropic response
    ↓
Start tools during streaming
    ↓
Append tool results
    ↓
Continue / compact / stop
```

## OpenCode: Session Prompt Loop + Session Processor

OpenCode's loop is centered on sessions, with a `SessionPrompt` orchestrator and a `SessionProcessor` that consumes model events.

```text
User prompt / command / API call
    ↓
Resolve session, agent, model, permissions, tools
    ↓
Build model messages from session history
    ↓
Assemble system prompt (provider prompt + env + skills + instructions)
    ↓
Stream LLM events
    ↓
SessionProcessor handles text / reasoning / tool calls / tool results
    ↓
Task or compaction parts may spawn child work
    ↓
Loop again until assistant stops
```

### Key traits

- built-in **agents** are part of the loop, not just prompt presets
- **subtasks** are represented as child sessions
- **compaction** can be inserted as a session task
- **permission requests** are part of normal tool execution
- the same loop can be driven from TUI, CLI run mode, HTTP server, web client, ACP, or SDK

## Pi: Session-Centered Event Loop

Pi centers the loop on `AgentSession`.

```text
Prompt / RPC command / SDK call
    ↓
Build session context
    ↓
Run extension hooks
    ↓
Stream provider response
    ↓
Execute tools through hookable pipeline
    ↓
Append tool results
    ↓
Continue or stop
```

## Hermes: AIAgent Conversation Loop

Hermes centers execution on one large `AIAgent` loop that branches across provider modes, callbacks, and tool execution.

```text
Prompt / gateway message / ACP request / cron task
    ↓
Resolve provider + API mode
    ↓
Build cached system prompt + ephemeral overlays
    ↓
Optionally pre-compress context
    ↓
Make interruptible model call
    ↓
Parse tool calls or final text
    ↓
Execute tools (single-threaded or thread pool)
    ↓
Append tool results / persist session / continue
```

### Key traits

- one **AIAgent** serves CLI, gateway, ACP, cron, and batch flows
- three API modes converge on one internal message format
- some tools are **intercepted at the agent level** (`todo`, `memory`, `session_search`, `delegate_task`)
- compression, retries, and fallback model activation live inside the main loop
- callbacks make the same loop observable from terminal UIs, messaging platforms, and editors

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Boundary** | Explicit turns | Generator loop | Session loop with processor | Session loop with event bus | Conversation loop with provider branches |
| **Primary state holder** | `TurnContext` | `AppState` | session store + processor state | `AgentSession` + `SessionManager` | `AIAgent` + conversation history + session DB |
| **Tool timing** | Spawn during stream, drain at turn end | Start during stream | Start from streamed events handled by processor | Start after assistant tool-call parts are materialized | Start after response parse; may execute concurrently |
| **Recovery path** | Retry + compaction | Retry + compaction cascade | Retry + compaction + pruning + subtask/session flow | Retry + compaction + branch summaries | Retry + compression + fallback model/provider + session persistence |
| **Client surfaces** | CLI-first | CLI/desktop product | TUI, web, attach, ACP, server, SDK | TUI, print, JSON, RPC, SDK | CLI, gateway, ACP, cron, batch, Python embedding |

## Why It Matters

Codex and Claude Code are mostly single-product loops.
OpenCode, Pi, and Hermes are more reusable, but in different ways:

- **OpenCode** exposes the loop through a product server and many clients.
- **Pi** exposes the loop through a smaller runtime and extension/SDK APIs.
- **Hermes** exposes one agent core through operational surfaces like messaging, cron, ACP, plugins, and different execution backends.
