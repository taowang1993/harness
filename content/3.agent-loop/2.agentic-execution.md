This document goes below the chapter summary and traces the execution mechanics of five systems: streaming, tool dispatch, retries, compaction, and continuation.

---

## The Common Pattern

All five systems implement a ReAct-style cycle:

```text
Observe → Think → Act → Observe → Think → Act → ... → Done
```

The differences are in:

- where state lives
- how tool calls are extracted and executed
- how retries and compaction work
- whether the runtime is product-owned, platform-oriented, or openly extensible

---

## Codex: Turn Loop with In-Flight Futures

Codex keeps a hard turn boundary.

```text
while needs_follow_up {
  build prompt
  stream model response
  spawn tool futures as tool items complete
  drain futures
  decide whether another turn is needed
}
```

Key traits:
- turn-scoped synchronization point
- read/write lock model for tool concurrency
- transport fallback between WebSocket and HTTP/SSE
- simpler compaction story than Claude Code

---

## Claude Code: Streaming Generator with Recovery Cascades

Claude Code streams assistant output and starts tools while the response is still arriving.

```text
while (true) {
  compact if needed
  stream assistant blocks
  on tool_use → start tool execution
  yield tool results
  continue / recover / stop
}
```

Key traits:
- `StreamingToolExecutor`
- many continuation and recovery sites
- strong model/output-limit recovery behavior
- deep compaction stack

## OpenCode: SessionPrompt + SessionProcessor

OpenCode splits the loop across two main layers:

- **`SessionPrompt.runLoop()`** decides what the session should do next
- **`SessionProcessor`** consumes streaming model events and turns them into session parts

### High-level flow

```text
while (true) {
  load session history
  inspect last user / assistant / task state
  if compaction task exists → compact
  if subtask exists → handle child session
  if assistant finished without pending tool calls → break

  create assistant message
  resolve tools for current agent/model/session
  build system prompt pieces
  stream LLM response
  SessionProcessor handles text/reasoning/tool events

  if processor says compact → enqueue compaction
  if processor says stop → break
  else continue
}
```

### What SessionProcessor does

It handles streamed events like:

- `text-start` / `text-delta` / `text-end`
- `reasoning-start` / `reasoning-delta` / `reasoning-end`
- `tool-call`
- `tool-result`
- `tool-error`
- `finish-step`

It also:
- updates session status
- creates patch/snapshot metadata for file changes
- detects overflow
- triggers permission requests and tool-state updates
- marks aborted tool executions as errors during cleanup

### Distinctive mechanics

#### 1. Subagents are child sessions

OpenCode's `task` tool does not spawn an OS process. It creates or resumes a **child session** and prompts a different agent there.

That means subagents are:
- isolated as sessions
- still inside the same overall runtime/server instance
- navigable from the UI via parent/child session controls

#### 2. Compaction is also modeled as session work

When context overflows, OpenCode can create a compaction task, run a hidden compaction agent, then continue the session with a summary.

It also prunes older tool outputs after the loop to reduce future context pressure.

#### 3. Permissions are in-band

Permission checks are not a separate daemon or classifier query. Tool calls directly invoke the permission system, which can:
- allow
- ask
- deny

and can persist "always allow" patterns for the running OpenCode session.

#### 4. The loop is server-native

Because OpenCode is built around a server, the same session loop can be driven from:
- TUI
- `run`
- `serve`
- `web`
- `attach`
- ACP clients
- SDK clients

## Pi: AgentSession Event Loop

Pi's loop is centered on `AgentSession`, with an event model shared across TUI, JSON, RPC, and SDK consumers.

### High-level flow

```text
prompt/steer/followUp
  ↓
rebuild session context
  ↓
run input / prompt / context hooks
  ↓
stream provider response
  ↓
tool_call hooks may mutate or block inputs
  ↓
execute tools
  ↓
tool_result hooks may rewrite outputs
  ↓
continue or stop
```

### Distinctive mechanics

- extensions can intercept the loop at multiple points
- tool execution emits public lifecycle events
- branch summarization is separate from ordinary compaction
- the runtime is intentionally smaller than OpenCode's product server architecture

## Hermes: AIAgent + Runtime Provider + Tool Dispatch

Hermes keeps most orchestration inside one large synchronous class, `AIAgent`, with helper modules around it.

### High-level flow

```text
while (true) {
  append user input / restore history
  build or reuse cached system prompt
  resolve runtime provider + api_mode
  pre-compress if context is getting large
  convert internal messages to provider format
  make interruptible API call

  if tool_calls returned {
    execute tools (inline or thread pool)
    append tool results
    continue
  }

  persist session + flush memory if needed
  return final response
}
```

### Distinctive mechanics

#### 1. Three provider modes converge on one internal format

Hermes supports:
- `chat_completions`
- `codex_responses`
- `anthropic_messages`

But before and after the provider call, it normalizes everything back to one OpenAI-style internal message shape. That keeps most of the loop provider-agnostic.

#### 2. API calls are interruptible

The actual provider request runs in a background thread while the main thread watches for:
- response completion
- user interrupt
- timeout

If interrupted, the in-flight result is discarded instead of being half-inserted into history.

#### 3. Some tools bypass normal registry dispatch

Most Hermes tools go through the registry, but a few are intercepted by the agent loop because they need direct access to agent state:
- `todo`
- `memory`
- `session_search`
- `delegate_task`

This makes the loop more product/platform-aware than a pure registry-only design.

#### 4. Compression, fallback, and persistence live in the same loop

Hermes does not treat context management as a separate service. The main loop can:
- preflight compress
- use gateway hygiene as a second safety net
- activate fallback model/provider chains
- save messages to SQLite sessions with lineage tracking

#### 5. Subagents are child AIAgent instances

Hermes delegation does not create OpenCode-style child sessions or Claude-style child OS processes by default. It spawns isolated child agent instances with:
- fresh conversations
- restricted toolsets
- separate terminal sessions
- only the final summary returned to the parent

## Comparison: Execution Techniques

| Technique | Codex | Claude Code | OpenCode | Pi | Hermes |
|-----------|-------|-------------|----------|----|--------|
| **Core loop shape** | turn loop | async generator | session loop + processor | session loop + event bus | synchronous agent loop + provider branches |
| **Tool extraction** | completed response items | streaming content blocks | streamed events into session parts | assistant tool-call parts | parsed response objects after provider call |
| **Subagent model** | in-process tasks | child processes | child sessions | not built in | child agent instances with fresh context |
| **Compaction style** | ghost snapshots | multi-layer compaction | hidden compaction agent + pruning | compaction entries + branch summaries | lossy summaries + gateway hygiene + pluggable context engine |
| **Permission integration** | approvals + policies | dialogs + classifier | permission rules in tool flow | user-defined via hooks/extensions | dangerous-command approval + gateway auth + backend choice |
| **Best fit** | contained shell agent | feature-rich product | open-source multi-client product | customizable harness runtime | operational agent platform with coding capabilities |

---

## Key Insight

OpenCode sits between Claude Code and Pi.
Hermes sits in a different spot again:

- like **OpenCode**, it has a broad product/runtime surface
- like **Pi**, it stays openly hackable and plugin-oriented
- unlike both, it also treats **messaging gateways, cron jobs, memory providers, and RL environments** as part of the same architecture

That makes Hermes especially useful to study if you want to understand what happens when an agent stops being just a coding CLI and becomes a broader agent platform.
