---
title: "File Editing"
description: "The editing primitives each system chooses, from exact replacement to multi-file patch workflows."
---

How an agent modifies code is one of the most consequential design choices in the system.

## Codex: Unified Patch Format

Codex uses a custom patch engine with multi-file transactional semantics.

## Claude Code: Single Replacement per Call

Claude Code uses exact string replacement with one file and one replacement per call.

## OpenCode: Exact Edit, MultiEdit, and Apply Patch

OpenCode exposes multiple editing styles.

### `edit`

- exact string replacement
- optional `replaceAll`
- permission-gated
- runs formatting after edits
- surfaces LSP diagnostics after modification

### `multiedit`

- sequentially applies multiple edit operations to the same file
- effectively gives OpenCode a built-in batched single-file edit path

### `apply_patch`

- multi-file patch application
- supports add, update, move, delete
- checks permissions across affected files
- returns per-file diff metadata

### Distinctive trait

OpenCode offers both **string replacement** and **patch-style** editing as first-class built-ins, rather than picking only one.

## Pi: Exact Replacement with `edits[]`

Pi lands between Claude Code and OpenCode.

- one file per call
- many disjoint replacements allowed in `edits[]`
- matches against the original file
- separate `write` for full-file writes
- per-file mutation queueing to avoid concurrent clobbers

## Hermes: `patch` + `write_file`

Hermes centers editing on a dedicated patch tool rather than a generic shell diff workflow.

### `patch`

- targeted find-and-replace editing
- fuzzy matching strategies tolerate minor whitespace/indentation drift
- returns a unified diff
- auto-runs syntax checks after editing
- designed to replace ad-hoc `sed`/`perl` terminal edits

### `write_file`

- full-file overwrite
- creates parent directories automatically
- paired with `read_file`/`search_files` for a structured edit loop

### Distinctive trait

Hermes is less diff-native than Codex and less modality-rich than OpenCode, but more editing-opinionated than shell-first systems: it wants the model to use explicit file tools, not terminal hacks.

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Patch support** | Yes | No | Yes | No | Yes |
| **Exact replacement** | Not primary | Yes | Yes | Yes | Yes, with fuzzy matching emphasis |
| **Multiple edits in one file** | Yes | No | Yes (`multiedit`) | Yes (`edits[]`) | Some targeted batching via patch workflows |
| **Multi-file edits** | Yes | No | Yes (`apply_patch`) | No | Not a major first-class path |
| **Post-edit diagnostics/formatting** | Varies | Some | Strong built-in integration | Simpler | Syntax-check oriented |

## Takeaway

- **Codex** is most diff-native.
- **Claude Code** is most rigidly string-replacement-native.
- **OpenCode** supports both styles as product features.
- **Pi** adds a smaller ergonomic middle ground without becoming a full editing subsystem.
- **Hermes** treats editing as a structured file-tool workflow, pushing the model away from shell one-liners and toward patch-oriented mutations.
