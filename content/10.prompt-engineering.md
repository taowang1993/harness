title: "Prompt Engineering"
description: "How the five systems assemble prompts from templates, files, overlays, and runtime metadata."

The system prompt is the agent's operating manual. Each system assembles it differently.

## Codex: Embedded Templates

Codex uses compile-time markdown templates plus runtime policy snippets.

## Claude Code: Dynamic Modular Sections

Claude Code builds a large runtime prompt from many sections and heavily optimizes for caching.

## OpenCode: Provider Prompts + Environment + Rules + Skills

OpenCode has a layered prompt assembly model.

### Main ingredients

- provider-specific base prompts (`anthropic`, `gpt`, `gemini`, `codex`, `kimi`, etc.)
- environment block: cwd, worktree, platform, git info
- `AGENTS.md` / `CLAUDE.md` / configured instruction files
- discovered skills rendered into the prompt and loaded through the `skill` tool
- agent-specific prompts and options
- plugin hooks that can transform system prompt pieces or the message list

### Distinctive trait

OpenCode combines a product-style prompt system with strong filesystem-driven customization. It is less static than Codex, more open than Claude Code, and less minimal than Pi.

## Pi: Resource-Driven Prompt Assembly

Pi is the most explicitly user-facing prompt system:
- `AGENTS.md`
- `SYSTEM.md`
- `APPEND_SYSTEM.md`
- skills
- tool prompt snippets
- extension hooks

## Hermes: Stable Prompt Layers + Ephemeral Overlays

Hermes makes a strong architectural distinction between **cached prompt state** and **API-call-time additions**.

### Main ingredients

- `SOUL.md` or default agent identity
- tool-use guidance
- `MEMORY.md` and `USER.md` snapshots
- skills index
- project context files such as `.hermes.md`, `AGENTS.md`, `CLAUDE.md`, `.cursorrules`
- timestamp, session ID, and platform hints
- ephemeral overlays for things like context pressure or gateway/session-specific hints

### Distinctive trait

Hermes is stability-first. It deliberately keeps long-lived prompt layers cached and frozen, while transient additions stay outside that stable prefix to preserve caching and predictable memory semantics.

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Base prompt shape** | Embedded templates | Modular runtime sections | Provider-specific prompt files | Runtime prompt + resources | Stable layered prompt + ephemeral overlays |
| **Filesystem instruction sources** | Some | Some | Strong | Strongest | Strong |
| **Skill integration** | Some | Some | Built-in skill tool | Built-in skill loading | Built-in skills index + skill tools |
| **Prompt mutation hooks** | Limited | Internal | Plugin hooks | Public extension hooks | More constrained, plugin/provider-oriented |
| **Customization stance** | Controlled | Product-owned | Product + open config/plugins | User-controlled by design | Open, but optimized for prompt stability |

## Takeaway

OpenCode, Pi, and Hermes are all highly customizable, but in different ways:
- **OpenCode** has a richer built-in prompt product with open hooks.
- **Pi** exposes the most obviously hackable prompt surface.
- **Hermes** exposes a layered prompt system that is designed to stay stable across long-running sessions and many runtime surfaces.
