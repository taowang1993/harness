All of these agents are model clients. They differ in provider breadth, transport, auth, and recovery.

## Codex: OpenAI-Centric Responses API Client

Codex is OpenAI-first, with SSE/WebSocket transport and some custom-provider support.

## Claude Code: Anthropic-Centric Messages API Client

Claude Code is Anthropic-first, with Bedrock and Vertex options layered on top.

## OpenCode: AI SDK + Models.dev + Product Server

OpenCode is deeply multi-provider.

### Characteristics

- built on the **AI SDK**
- uses **Models.dev** for a broad model catalog
- supports 75+ providers plus local/OpenAI-compatible endpoints
- handles auth through `/connect` and credential storage
- supports custom providers through config
- exposes the runtime over **HTTP/OpenAPI**, **SDK**, and **ACP**
- includes a dedicated `small_model` path for lightweight tasks like title generation

### Transport and runtime shape

OpenCode is not just a model client inside a CLI. It is a server-backed runtime that can be driven by multiple clients.

## Pi: Model Registry + Provider Abstraction

Pi is also deeply multi-provider, but with a smaller, more harness-like surface.

### Characteristics

- `AuthStorage`
- `ModelRegistry`
- many built-in providers
- `models.json` and `registerProvider()` for custom providers
- CLI, RPC, JSON mode, and SDK all share the same model/runtime layer

## Hermes: Shared Runtime Resolver + Three API Modes

Hermes also leans hard into provider abstraction, but does it through one shared runtime resolver used by every surface.

### Characteristics

- a shared resolver maps `(provider, model)` to `api_mode`, `base_url`, `api_key`, and source metadata
- supports many built-in providers plus custom OpenAI-compatible endpoints
- has three first-class execution modes: `chat_completions`, `codex_responses`, `anthropic_messages`
- auxiliary tasks can route to their own provider/model independently of the main chat model
- fallback model/provider chains are built into the runtime
- the same runtime logic is reused by CLI, gateway, cron, ACP, and helper tasks

### Transport and runtime shape

Hermes is not server-first in the same way as OpenCode, but it is still a multi-surface runtime. Provider resolution is a shared subsystem rather than a CLI-local concern.

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Provider bias** | OpenAI-first | Anthropic-first | Strongly multi-provider | Strongly multi-provider | Strongly multi-provider |
| **Client surfaces** | CLI-focused | Product-focused | CLI + server + web + ACP + SDK | CLI + RPC + SDK | CLI + gateway + ACP + cron + Python library |
| **Custom providers** | Some | Limited | Yes | Yes | Yes |
| **OpenAPI/SDK story** | Limited | Limited | Strong | SDK/RPC focused | Python embedding + gateway/API/ACP surfaces |
| **Local model friendliness** | Good | Moderate | Strong | Strong | Strong |

## Takeaway

OpenCode, Pi, and Hermes are all provider-agnostic, but they optimize differently:
- **OpenCode** is a multi-provider coding product.
- **Pi** is a multi-provider coding harness.
- **Hermes** is a multi-provider agent platform that keeps provider resolution shared across operational surfaces like chat, gateway, cron, and ACP.
