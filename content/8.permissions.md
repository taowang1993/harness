Permissions decide what the agent can do automatically and what requires human approval.

## Codex: Guardian Approval System

Codex uses typed approval requests and execution policy matching.

## Claude Code: Multi-Layer Permission Stack

Claude Code combines rule matching, product dialogs, and an LLM classifier.

## OpenCode: Pattern-Based Permission Engine

OpenCode has one of the clearest built-in permission systems in this comparison.

### Actions

Each rule resolves to:
- `allow`
- `ask`
- `deny`

### What can be controlled

- `read`
- `edit` (covers `edit`, `write`, `apply_patch`, `multiedit`)
- `glob`
- `grep`
- `list`
- `bash`
- `task`
- `skill`
- `lsp`
- `question`
- `webfetch`, `websearch`, `codesearch`
- `external_directory`
- `doom_loop`

### Distinctive features

- object syntax for granular command/path matching
- wildcard patterns with last-match-wins semantics
- `once` / `always` / `reject` approval outcomes
- per-agent permission overrides
- permissive defaults, but `external_directory` and `doom_loop` ask by default
- `.env` reads denied by default

## Pi: No Default Permission Subsystem

Pi has no built-in permission model. You build one yourself with extensions, custom tools, or the environment it runs inside.

## Hermes: Dangerous-Command Approvals + Gateway Auth

Hermes has a narrower but very real built-in permission product.

### Actions

Command approvals can resolve to:
- `once`
- `session`
- `always`
- `deny`

And gateway access can independently:
- allow a user
- deny a user
- require DM pairing first

### What can be controlled

- dangerous terminal commands that match curated regex patterns
- session or permanent command allowlisting
- who can talk to the gateway on each messaging platform
- whether unauthorized DMs trigger pairing or are ignored
- whether smart approval uses an auxiliary model to auto-approve / auto-deny

### Distinctive features

- approval is strongest around **terminal risk**, not a universal per-tool matrix
- `smart` mode adds a classifier-like auxiliary LLM decision layer
- container/cloud backends shift security responsibility from approvals to isolation
- gateway auth and agent execution permissions live in the same broader security model

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Built-in approval system** | Yes | Yes | Yes | No | Yes |
| **Granular path/command patterns** | Yes | Yes | Yes | Only if user builds them | Command-pattern focused; less universal than OpenCode |
| **Classifier-based approval** | No | Yes | No | No | Optional smart-approval path |
| **Per-agent overrides** | Limited | Some | Yes | User-defined | Not central |
| **Session-persistent allow rules** | Yes | Yes | Yes | User-defined | Yes |

## Key Difference

Claude Code is the most classifier-heavy.
OpenCode is the most explicitly **pattern-and-agent-config driven**.
Pi delegates the entire question to the user.
Hermes focuses permissions more narrowly on terminal safety and gateway authorization, then relies on backend isolation to carry the rest.
