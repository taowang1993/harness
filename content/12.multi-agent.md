---
title: "Multi-Agent Architecture"
description: "How sub-agents are isolated, coordinated, and surfaced across different coding-agent products."
---

Sub-agents are one of the clearest places where product philosophy shows up.

## Codex: In-Process Agent Tree

Codex has built-in sub-agents running as async tasks with shared process state.

## Claude Code: Separate Child Processes

Claude Code has built-in sub-agents running as child OS processes.

## OpenCode: Child Sessions via the `task` Tool

OpenCode has a built-in multi-agent model, but it is neither Codex-style async tasks nor Claude-style child processes.

### Characteristics

- primary agents like `build` and `plan`
- subagents like `general` and `explore`
- `task` tool launches or resumes **child sessions**
- task permissions control which agents another agent may invoke
- child sessions can be navigated from the UI as part of the session graph
- the whole system runs inside the same overall runtime/server instance

### Tradeoff

This gives OpenCode stronger built-in agent orchestration than Pi, but less hard isolation than Claude Code's process model.

## Pi: No Built-In Sub-Agents by Design

Pi intentionally omits built-in sub-agents.

Instead you can:
- run multiple pi instances
- use tmux
- spawn sessions from the SDK
- build a sub-agent tool or workflow with extensions

## Hermes: `delegate_task` Child Agents

Hermes has a built-in delegation model centered on isolated child agent instances.

### Characteristics

- `delegate_task` spawns child `AIAgent` instances
- each child gets a **fresh conversation**, not inherited history
- each child gets its **own terminal session** and restricted toolsets
- only the final summary flows back into the parent context
- up to 3 children can run concurrently in common batch mode
- depth is intentionally limited and several tools are always blocked for children

### Tradeoff

Hermes gives you stronger built-in delegation than Pi and stronger context isolation than OpenCode's child-session model, but it is still less isolated than Claude Code's child-process architecture.

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Built-in sub-agents** | Yes | Yes | Yes | No | Yes |
| **Isolation model** | async tasks | child processes | child sessions in same runtime | user-defined | child agent instances with fresh context and separate terminals |
| **History sharing** | forked parent history | fresh task prompt | separate child session with inherited context rules | user-defined | explicit `goal`/`context` only |
| **Navigation** | internal | limited | explicit child-session navigation | user-defined | summary/progress oriented, not session-graph oriented |
| **Opinionation** | High | High | High | Deliberately low | High |

## Takeaway

OpenCode is the open-source system here with the most built-in multi-agent product surface.
Pi is the one that most strongly refuses to choose a sub-agent model for you.
Hermes is notable because it chooses a very practical middle ground: built-in delegation with fresh-context child agents and separate terminals, but without turning multi-agent work into a full session-graph UI.
