## Project Structure

### Codex CLI — Rust Workspace

Codex is a Rust workspace with clear crate boundaries.

```text
codex/
├── codex-cli/              # npm wrapper
└── codex-rs/
    ├── cli/                # CLI entry
    ├── tui/                # Ratatui UI
    ├── core/               # agent loop, tools, approvals
    ├── sandboxing/         # sandbox abstraction
    ├── execpolicy/         # execution policy
    └── ...
```

### Claude Code — TypeScript Product Monolith

Claude Code is one large TypeScript product organized by feature.

```text
claude-code-source-code/
└── src/
    ├── main.tsx
    ├── query.ts
    ├── QueryEngine.ts
    ├── tools/
    ├── services/
    ├── components/
    ├── state/
    └── utils/
```

### OpenCode — Bun Monorepo with Shared Runtime

OpenCode is a TypeScript monorepo with a central runtime package and multiple clients.

```text
opencode/
└── packages/
    ├── opencode/           # core CLI/runtime/server package
    ├── app/                # web app
    ├── desktop/            # desktop client
    ├── sdk/js/             # JS SDK
    ├── plugin/             # plugin helpers/types
    ├── ui/                 # shared UI pieces
    └── web/                # documentation site
```

Important design characteristics:

- the **server is first-class**, not an afterthought
- the TUI is only one client of the runtime
- the runtime also supports **web**, **IDE/ACP**, **SDK**, and **HTTP/OpenAPI** integrations
- plugins, custom tools, MCP servers, and agents are part of the core architecture

### Pi — Layered TypeScript Monorepo

Pi is a layered monorepo with reusable packages beneath the CLI.

```text
pi-mono/
└── packages/
    ├── ai/                 # provider abstraction
    ├── agent/              # generic agent runtime
    ├── tui/                # terminal UI components
    └── coding-agent/       # CLI, SDK, tools, modes
```

### Hermes — Python Agent Platform Repository

Hermes is a Python repo organized around one large agent core plus multiple runtime surfaces and plugin systems.

```text
hermes-agent/
├── run_agent.py            # AIAgent core loop
├── cli.py                  # terminal UI
├── model_tools.py          # tool discovery and dispatch
├── toolsets.py             # tool grouping and presets
├── hermes_state.py         # SQLite session storage
├── agent/                  # prompt, compression, providers, memory, trajectories
├── hermes_cli/             # CLI subcommands, auth, model selection, plugins
├── tools/                  # self-registering tool modules + environments
├── gateway/                # messaging platform gateway
├── acp_adapter/            # editor/ACP server
├── cron/                   # scheduled jobs
├── environments/           # RL/eval environments
├── plugins/                # memory and context-engine plugins
└── website/                # docs
```

Important design characteristics:

- the same **AIAgent** is reused across CLI, gateway, ACP, cron, and batch/eval paths
- toolsets, plugins, memory providers, and context engines are all first-class architectural concepts
- terminal execution can target **local**, **Docker**, **SSH**, **Singularity**, **Modal**, or **Daytona** backends
- Hermes is broader than a coding product: it is an agent runtime, messaging system, and experimentation platform in one repo

## Entry Points

### Codex

One binary, many subcommands:

```text
codex
codex exec
codex review
codex mcp-server
codex app-server
codex resume
```

### Claude Code

One main app with fast paths for utility commands and daemon modes.

### OpenCode

OpenCode has a wider set of first-class entry points:

```text
opencode                 → TUI (and local server)
opencode run             → non-interactive execution
opencode serve           → headless HTTP server
opencode web             → headless server + browser UI
opencode attach          → attach TUI to existing server
opencode acp             → Agent Client Protocol server
opencode auth / mcp / session / models / stats / export / import
```

This is a major architectural difference from Codex and Claude Code: OpenCode is designed from the start as a **server-backed multi-client system**.

### Pi

Pi is also multi-mode, but with a smaller surface:

```text
pi                    → interactive mode
pi -p                 → print mode
pi --mode json        → JSON event stream
pi --mode rpc         → RPC over stdin/stdout
```

### Hermes

Hermes exposes several operational entry points around the same agent core:

```text
hermes                 → interactive CLI
hermes chat            → scripted/non-interactive chat flow
hermes gateway         → long-running messaging gateway
hermes acp             → ACP server for editors
batch_runner.py        → trajectory/data generation
API server / Python library paths → embedded agent usage
```

This makes Hermes less like a single coding CLI and more like an **agent platform with multiple runtime shells**.

## Build & Distribution

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Build tool** | Cargo | Bun/native installers | Bun | `tsgo` + optional Bun binary | Python packaging (`uv`/`pip`) |
| **Primary output** | Native binary | Native app / CLI | CLI + desktop + server + web + SDK | CLI + SDK | CLI + gateway + ACP + cron + env framework |
| **Runtime style** | Native | Product runtime | Product runtime with server core | Harness runtime | Agent platform runtime |
| **Distribution** | npm wrapper + binaries | Installers / desktop app | install script, npm, Homebrew, desktop builds | npm | repo install / Python package |
| **Open source** | Yes | No | Yes | Yes | Yes |

## Architectural Takeaways

### Codex optimizes for systems clarity

The Rust workspace cleanly separates protocol, core, sandboxing, and policies.

### Claude Code optimizes for product integration

The codebase is less neatly separated, but the product surface is very rich.

### OpenCode optimizes for product breadth without provider lock-in

OpenCode is an open-source product architecture: built-in server, multiple clients, plugins, MCP, built-in agents, and provider abstraction all live together.

### Pi optimizes for reuse and modification

Pi is the least product-like of the group. The architecture is designed so the runtime can be embedded, extended, or reshaped rather than treated as a fixed appliance.

### Hermes optimizes for operational breadth

Hermes is the most platform-like architecture here. It reuses one agent core across chat, gateway, ACP, cron, plugins, memory, and even RL/eval environments, with a lot of practical runtime machinery living beside the coding-agent loop.
