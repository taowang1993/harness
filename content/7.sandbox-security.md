Security is where the systems diverge most sharply.

## Codex: Kernel-Level Sandboxing

Codex wraps execution in OS-level sandboxing.

- macOS: Seatbelt
- Linux: Landlock + seccomp + bubblewrap
- Windows: restricted tokens

This gives Codex the strongest built-in containment story.

## Claude Code: Application-Level Controls

Claude Code has no kernel sandbox. Instead it uses path rules, dangerous-pattern checks, and an LLM classifier before execution.

## OpenCode: Application-Level Permissions with Workspace Guards

OpenCode also does not ship a kernel sandbox. Its safety model is application-level, but more rule-driven than classifier-driven.

### Built-in guard rails

- per-tool permissions: `allow`, `ask`, `deny`
- granular pattern matching for bash commands, file paths, and skill names
- `external_directory` approvals for paths outside the workspace
- `doom_loop` approvals when the same tool call repeats
- per-agent permission overrides
- `.env` reads denied by default

OpenCode is still not a hard sandbox, but it has a substantial built-in permission product.

## Pi: Bring Your Own Isolation

Pi intentionally does not ship a built-in sandbox or permission popup system.

The expected approaches are:
- run pi inside a container, VM, remote host, or SSH environment
- intercept tools with extensions
- replace built-ins with safer variants
- add your own confirmations and policies

## Hermes: Hybrid Safety Model

Hermes combines application-level approval logic with optional execution isolation.

### Built-in guard rails

- dangerous-command detection and approval prompts on local/SSH execution
- approval modes: `manual`, `smart`, `off`/YOLO
- gateway user authorization via allowlists and DM pairing
- context-file prompt-injection scanning
- environment filtering for child processes, MCP, and code execution
- optional isolated backends: Docker, Singularity, Modal, Daytona, or SSH

A key Hermes idea is that **the active terminal backend changes the safety model**:
- on `local` and `ssh`, dangerous commands are checked and may require approval
- in container/cloud backends, dangerous-command prompts are skipped because the sandbox is treated as the main boundary

## Comparison

| Aspect | Codex | Claude Code | OpenCode | Pi | Hermes |
|--------|-------|-------------|----------|----|--------|
| **Kernel sandbox** | Yes | No | No | No | No |
| **Built-in permission UX** | Yes | Yes | Yes | No | Yes |
| **Hard containment** | Strongest | Weak | Weak | External only | Optional/conditional via backend choice |
| **Rule-driven app controls** | Yes | Yes | Yes | Only if user adds them | Yes |
| **Extensible policy hooks** | Some | Limited | Plugins and config | Extensions and SDK | Plugins + backend selection + config |

## Takeaway

- **Codex** is strongest on built-in containment.
- **Claude Code** is strongest on productized approval UX.
- **OpenCode** offers a strong open-source application-layer permission system.
- **Pi** gives you the least built-in protection and the most responsibility.
- **Hermes** offers the most hybrid story: app-layer approvals by default, with real isolation available when you move execution into containers, remote hosts, or cloud sandboxes.
