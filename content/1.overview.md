## What Is a Coding Agent?

A coding agent is an AI system that can autonomously read code, run tools, edit files, and iterate toward a goal. Unlike a chatbot that only suggests code, a coding agent **acts** inside a workspace.

The basic loop still looks simple:

```text
User prompt → LLM reasoning → Tool calls → Results fed back → LLM continues → ... → Done
```

The hard part is everything around that loop: permissions, context management, sandboxing, UX, recovery, extensibility, and cost control.

## The Five Agents

This project compares five agent systems that each make a different architectural bet.

### Codex CLI (OpenAI)

- **Repository**: [openai/codex](https://github.com/openai/codex)
- **Language**: Rust
- **UI**: Ratatui terminal UI
- **Distribution**: Native binaries via npm wrapper
- **Core thesis**: strong containment around a shell-first agent

### Claude Code (Anthropic)

- **Source**: Closed-source; analysis based on [ultraworkers/claw-code](https://github.com/ultraworkers/claw-code) (v2.1.88)
- **Language**: TypeScript
- **UI**: Custom Ink-like terminal renderer
- **Distribution**: Native installers and desktop app
- **Core thesis**: rich product logic with application-level controls

### OpenCode (Anomaly)

- **Repository**: [anomalyco/opencode](https://github.com/anomalyco/opencode)
- **Language**: TypeScript
- **UI**: Terminal UI, desktop app, web client, IDE integrations
- **Distribution**: install script, package managers, desktop releases
- **Core thesis**: provider-agnostic, open-source coding product with a client/server architecture

### Pi (Mario Zechner)

- **Repository**: [badlogic/pi-mono](https://github.com/badlogic/pi-mono/tree/main/packages/coding-agent)
- **Language**: TypeScript
- **UI**: `pi-tui`
- **Distribution**: npm package and SDK
- **Core thesis**: minimal harness with first-class extensibility

### Hermes (Nous Research)

- **Repository**: [nousresearch/hermes-agent](https://github.com/nousresearch/hermes-agent)
- **Language**: Python
- **UI**: terminal CLI, messaging gateway, ACP/IDE bridge
- **Distribution**: Python package / repo install
- **Core thesis**: a broad agent platform where tools, gateways, plugins, memories, and execution backends are first-class

## Design Philosophies

### Codex: "Sandbox everything, then trust the shell"

Codex relies on kernel-level sandboxing and keeps the tool surface relatively small. The shell is the main universal interface.

### Claude Code: "Analyze everything, ask before risky actions"

Claude Code puts more intelligence into the application layer: many specialized tools, permission rules, classifiers, and recovery mechanisms.

### OpenCode: "Provider-agnostic product with open interfaces"

OpenCode is closer to Claude Code than to Pi in product ambition: it ships a real multi-surface experience, built-in agents, permissions, MCP, plugins, a server, a web client, ACP support, and an SDK. But unlike Claude Code, it is open source and deliberately provider-agnostic.

### Pi: "Keep the core small, let users shape the workflow"

Pi intentionally omits many product features so users can add them themselves with extensions, skills, prompt templates, packages, RPC, or the SDK.

### Hermes: "Treat the agent as a platform, not just a coding shell"

Hermes is broader than a coding-only CLI. It keeps one agent core, then fans it out through a terminal UI, messaging gateway, ACP, cron jobs, plugins, memory providers, context engines, and multiple execution backends.

## The Tradeoff

| Dimension | Codex | Claude Code | OpenCode | Pi | Hermes |
|-----------|-------|-------------|----------|----|--------|
| Core bet | Shell inside hard containment | Rich app-layer control | Open-source, provider-agnostic coding product | Minimal, extensible harness | Broad agent platform with gateways, plugins, and backend flexibility |
| Default safety story | Kernel sandbox | App permissions + classifier | App permissions + patterns | External environment or user extensions | Dangerous-command approvals + optional isolated backends |
| Product surface | Focused | Broad | Broad | Deliberately small | Broadest overall surface in this set |
| Provider strategy | OpenAI-first | Anthropic-first | Strongly multi-provider | Strongly multi-provider | Strongly multi-provider |
| Extensibility | Some | Moderate | Strong plugins/MCP/custom tools | Strongest raw customization surface | Strong plugins + provider plugins + runtime backends |
